# 評価フレームワーク詳細

このファイルは AI協働プロフィールの5軸+統制レイヤーの詳細な評価基準を定義する。
分析時にこのファイルを読み込み、各軸の観測基準に従って採点する。

---

## 設計思想

このスキルは、AIをただ制限する力ではなく、
**AIに必要な文脈を与え、対話を操舵し、必要な場面だけ統制し、最後に見極めて採用する力** を評価する。

評価対象は「言い回し」ではなく、4つの空間の設計を見る:

- **問題空間**: 何を解くのか（→ 目的設計力）
- **文脈空間**: 何を見せるのか（→ 文脈供給力）
- **探索空間**: どこまで広げて、どこで絞るのか（→ 対話操舵力）
- **行動・判断空間**: どこまで任せ、どこで止め、何を採るのか（→ 検証・採択力 + 統制レイヤー）

4つの空間 + メタレベル（学習資産化力）できれいに対応する。

止める力は衛生要因（ないと問題だが、あるだけでは差がつかない）。
文脈を渡す力、広げる・絞る・切り替える力がレバレッジ要因（ここで差がつく）。

---

## 主プロフィール5軸

### 1. 目的設計力（Purpose Design）

何を作るのか、誰向けか、何をもって成功とするかを定める力。

#### 見ること
- 何を作る / 解くのかが見えているか
- 誰向けか、何のためかが示されているか
- 完了条件や優先順位があるか
- 曖昧なタスクでも、目的を徐々に定めにいけているか
- 手段ではなく成果基準に触れているか

#### 観測シグナル

高い目的設計力のシグナル:
- 「〜な人向けに、〜の目的で」と対象と意図を明示
- 「成功基準は〜」「最低限〜ができれば」と完了条件を定義
- 曖昧な依頼でも途中で「要するに〜がしたい」と再定義
- 複数のゴールがある場合に優先順位を付けている

低い目的設計力のシグナル:
- 「なんかいい感じにして」で目的不在
- 手段だけを指定（「React使って」）して何を作るかがない
- 途中で目的がブレて収拾がつかなくなる
- AIが「何を作りましょうか？」と聞き返している

#### 減点しないこと
- 短文であること自体（短くても目的が明確なら高評価）
- いきなり完璧な定義がないこと自体（徐々に定めていればOK）
- プロトタイプ段階で完了条件が緩いこと

---

### 2. 文脈供給力（Context Supply）

背景、制約、素材、例、好み、避けたいことを、必要なタイミングで渡す力。

#### 見ること
- 背景情報を渡しているか（なぜこれが必要か）
- 制約を渡しているか（技術的制約、時間的制約、予算、対象環境）
- 素材を渡しているか（参考資料、既存コード、データ）
- 例を渡しているか（こういう感じ、このサイトみたいに）
- 好みを渡しているか（トーン、スタイル、フォーマット）
- 避けたいことを渡しているか（これはやらないで、この方向は不要）
- 必要な情報が**必要なタイミング**で入っているか
- AIが迷いやすい箇所に追加文脈を入れているか
- 足りない文脈を後から適切に補えているか

#### 観測シグナル

高い文脈供給力のシグナル:
- 最初の依頼に背景・対象・制約が含まれている
- AIが誤解しそうな箇所で先回りして補足
- 「参考にしてほしいのはこれ」と素材を提示
- 「〜は避けたい」と除外条件を明示
- AIの出力がズレたとき「足りなかった情報はこれ」と補足

低い文脈供給力のシグナル:
- 「〜作って」だけで背景ゼロ
- AIが何度も確認質問を投げている
- 後出しで大幅な方向転換（最初に言えた情報を後から追加）
- 「さっきのやつ」「あれ」で指示対象が不明確

#### 重視する点
- 情報量の多さではなく、**適切さとタイミング**
- 長文を書けるかではなく、必要な情報が入っているか

#### 減点しないこと
- 長文でないこと自体
- 専門用語が少ないこと自体
- 対話の中で徐々に補足していること（むしろ良い）

---

### 3. 対話操舵力（Dialogue Steering）— 中核指標

探索空間全体を操舵する力。事前に広げる（発散）も、事後に絞る（収束）も、この軸で見る。
5軸の中で最も重要な指標。

#### 見ること
- **発散**: 視点を広げる、複数案を出させる、別解を引き出す
  - 「3案出して」「別のアプローチも見せて」「反対意見は？」
  - AIを「実行代行」だけでなく、思考の探索機として使えているか
- **収束・修正**: AIの返答を受けて方向を変えているか
  - 広げる: 「それに加えて〜も考えて」
  - 狭める: 「〜に絞って」「この部分だけ」
  - 比較する: 「AとBどちらがいい？」「さっきの案とこの案を比較して」
  - 修正する: 「ここが違う、〜にして」
  - 前提を言い直す: 「前提が違った、実は〜」
  - ピボットする: 「方向を変えよう、〜にしたい」
- AIのズレや弱さを受けて、立て直しができているか
- 一度の返答で満足せず、必要な修正をかけているか
- AIの返答の**良い部分を残して悪い部分だけ直す**ことができているか

#### 観測シグナル

高い対話操舵力のシグナル:
- 「3つの案を比較したい」「反対意見も聞かせて」（発散）
- 「別の技術スタックだとどうなる？」「メリット・デメリットを並べて」（探索の幅出し）
- 「方向性はいいけど、〜の部分をもっと〜にして」（部分修正）
- 「ちょっと待って、前提が変わった」（前提の再設定）
- 「これは良いけど、別の角度からも見たい」（追加探索の要求）
- 「3つのうちAとCを組み合わせて」（統合）
- AIがズレた回答をした後の的確な軌道修正

低い対話操舵力のシグナル:
- 常に1案で進行し、AIの最初の提案をそのまま採用するパターンが支配的
- 「違う」だけで何が違うか伝えない
- ズレた回答に対して最初からやり直しを指示（部分修正ができない）
- 会話が一問一答で終わる（往復がない）

#### 重視する点
- ここは中核指標。特に **初回回答が弱いときのハンドリング** を重く見る
- 方向転換のうまさ、修正の的確さを評価

#### データソースによる制約
自動収集（Claude Code, Copilot, Cline, Roo, Windsurf, Antigravity）はユーザーメッセージのみを収集する。AIの返答が不明な状態での操舵力評価は間接的な推測に依存するため、confidence を「低」または「中」に制限する。手動インポート（ChatGPT, Claude.ai, Gemini エクスポート）は両方のメッセージを含むため、より正確な評価が可能。

#### 減点しないこと
- AIの一発目がたまたま良くて追加操舵が少ないこと
- 単純作業（変換系、作業系）で探索・操舵の機会自体が少ないこと
- すでに答えが見えているタスクで発散が少ないこと
- 短期間のデータで探索の機会自体が少ないこと

### 4. 検証・採択力（Verification & Adoption）

AIの提案をそのまま通すのではなく、比べる、確かめる、捨てる、採る、統合する力。

#### 見ること
- AIの提案をそのまま通していないか
- 比較をしているか（「AとBどちらが〜」）
- 根拠確認をしているか（「なぜそうなる？」「ソースは？」）
- 事実確認をしているか（「それ本当？」「最新の情報？」）
- レビューをしているか（出力を読んで問題を指摘）
- 採用 / 修正 / 保留 / 棄却を自分で決めているか
- 複数案を統合しているか

#### 観測シグナル

高い検証・採択力のシグナル:
- 「これで合ってる？ 〜の観点で確認して」
- 「根拠を教えて」
- 「この部分は採用、この部分は修正して」
- 「前の案の方がよかった、戻して」
- AIの出力に対して自分で問題を見つけて指摘

低い検証・採択力のシグナル:
- AIの出力を常に「それでOK」で通す
- 明らかに怪しい回答もそのまま受け入れる
- 「全部いい感じにして」で丸投げ
- 最終成果物を自分でレビューしている形跡がない

#### 減点しないこと
- 低リスクな軽作業で毎回厳密な検証をしていないこと
- 信頼できるAIの出力を効率的に採用していること（過度な検証は非効率）

---

### 5. 学習資産化力（Learning Asset Creation）

うまくいった進め方を、次回使えるテンプレートやチェックリスト、設定、仕組みに変える力。
他の4軸が「1回の対話の巧さ」なのに対し、この軸は **AI協働環境を自分で進化させているか** というメタレベルの力を見る。

#### 見ること

エンジニア向けツール（Claude Code等）での観測対象:
- CLAUDE.md へのルール・知見の書き込み
- Skills の作成・改善（`/skill-creator` 等）
- サブエージェントの設計・役割分担の定義
- プロンプトテンプレートの定型化（同じ構造の指示を再利用）
- hooks やワークフローの自動化設定
- 「次からはこうする」「このパターンを覚えて」等の知見の明示的な言語化
- `.claude/` 配下への再利用可能な資産の蓄積

非エンジニア向けツール（ChatGPT, Claude.ai等）での観測対象:
- Custom Instructions の設定・更新
- GPTs（カスタムGPT）の作成
- Project Knowledge / System Prompt への追記
- よく使うプロンプトのテンプレート化
- 過去のうまくいった指示を再利用している形跡
- 「前にうまくいったやり方で」と過去の成功パターンに言及

#### 観測シグナル

高い学習資産化力のシグナル:
- 「CLAUDE.md に追加して」
- 「これをスキルにして」「テンプレート化して」
- 「次からはこのルールで」
- 「前回のパターンを再利用して」
- 同じ種類の作業で回を追うごとに指示が洗練されている

低い学習資産化力のシグナル:
- 毎回同じ種類の作業をゼロから説明し直している
- 過去の成功パターンへの参照がない
- 設定やカスタマイズの痕跡がない

#### 減点しないこと
- 短期間のデータで再利用痕跡が少ないこと
- ツールが学習資産化機能を持たない場合（Custom Instructions がないツール等）
- 暗黙的に学んでいるが明示化していないこと（ただし confidence: 低 にする）

## 統制レイヤー（Control Layer）

主プロフィールとは**別建て**で評価する。
高リスク・不可逆・外部影響のある場面でのみ強く見る。

#### なぜ別建てにするか
- すべての対話で高リスク行為が発生するわけではない
- 統制が必要な場面がなかった場合に低スコアにするのは不公平
- 「止める力」は衛生要因であり、主プロフィールの中心ではない

#### 見ること
- AIに任せる範囲を調整しているか（「ここまでは任せるが、〜は確認させて」）
- 制約や禁則を渡しているか（「〜はしないで」「〜は変えないで」）
- 公開、送信、削除、上書き、共有、購入、デプロイなどの前に確認点を置いているか
- 個人情報、社外秘、シークレット、認証情報の扱いに注意しているか
- AIの計画が危ういときに止めたり、再計画させたりしているか
- 出力の粒度を調整しているか（「まず概要だけ見せて」→ 確認 → 「詳細に進んで」）

#### 判定基準

| 判定 | 意味 |
|------|------|
| 該当場面なし | 分析範囲に高リスク場面が含まれなかった。評価対象外。 |
| 良好 | 高リスク場面で適切に委任境界と承認点を設計していた。 |
| 要注意 | 高リスク場面で確認が不足していた、または制約が甘かった。 |
| 重大注意 | 高リスク行為をAIに無条件で委任、シークレット漏洩等の深刻な問題があった。 |

#### 高リスク場面の例
- git push / デプロイ / 本番環境操作
- メール送信 / Slack投稿 / SNS投稿
- ファイル削除 / データベース操作
- 課金・購入・契約に関わる操作
- 個人情報・クレデンシャルを含む処理
- 外部APIへの書き込みリクエスト

## スコアリング基準

### 0〜4 スケール

| スコア | ラベル | 判定の目安 |
|--------|--------|-----------|
| 0 | 未観測 | データ内にこの軸を判定できる証拠がない。データ不足の場合もこれ。 |
| 1 | 芽がある | 意識の片鱗は見えるが、散発的で一貫性がない。1〜2件の証拠。 |
| 2 | 時々できている | 一部の場面では発揮されているが、場面によってばらつきがある。 |
| 3 | 安定している | 多くの場面で一貫して発揮されている。明確なパターンがある。 |
| 4 | 他者の参考になる | 優れたパターンとして他の人に共有できるレベル。具体的な好事例がある。 |

### confidence の判定

| confidence | 条件 |
|-----------|------|
| 高 | 該当軸の証拠が5件以上あり、パターンが明確 |
| 中 | 証拠が2〜4件、またはパターンにばらつきがある |
| 低 | 証拠が1件以下、または間接的な推測に依存 |

## 仕事タイプ別の評価補正

仕事タイプによって「その軸が低くて当然」なケースがある。
不当な減点を避けるために、以下の補正を意識する。

| 仕事タイプ | 低くても問題ない軸 | 高いと特に評価すべき軸 |
|-----------|-------------------|---------------------|
| 生成系 | — | 目的設計力、文脈供給力 |
| 分析系 | 対話操舵力（分析対象が決まっている場合） | 検証・採択力 |
| 変換系 | 対話操舵力 | 文脈供給力（変換仕様の明確さ） |
| 対話系 | 検証・採択力（正解がない場面） | 対話操舵力 |
| 作業系 | 対話操舵力 | 目的設計力（タスク定義の明確さ） |
